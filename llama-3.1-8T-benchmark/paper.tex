\subsubsection{Large Language Model}
% Nil
We benchmark LLM inference using Llama-3.1-8B-Instruct \cite{llama3.1-8b}, the
instruction-tuned 8B variant of Meta's Llama~3.1 family (July~2024), across a
single NVIDIA A100, a single Intel Gaudi2 HPU, a single NVIDIA H100, and a single
NVIDIA GH200 Grace Hopper Superchip. At 8~billion parameters it is large enough to
stress accelerator memory bandwidth and compute, yet small enough to run on one
device on every platform, enabling controlled hardware comparisons free of
tensor-parallel or interconnect confounds. It is also an established accelerator
benchmark, used as a Gaudi2 baseline by Intel \cite{intel2024gaudi2} and in the
MLPerf Training suite \cite{mlcommons}.

The serving engine differs by platform, and we control for this confound rather
than eliminate it: each platform runs a well-supported software path whose exact
framework and version are recorded per run. The three NVIDIA platforms use vLLM on
its CUDA backend over the PyTorch release matching the installed CUDA toolkit
\cite{pytorch}. The vLLM Habana extension could not be brought up reliably on our
stack, so Gaudi2 instead uses Intel's Optimum-Habana inference stack in lazy mode
with HPU graphs—the vendor-preferred high-performance path. Graph capture and
compilation occur in a warmup phase excluded from timing on every platform, so
compilation latency is never counted as inference. vLLM separates prefill (time to
first token) from decode (inter-token latency), whereas Optimum-Habana reports a
single aggregate throughput; Gaudi2 therefore contributes to the throughput and
energy comparisons but not to the latency-decomposition (TTFT/ITL) results.

Precision is treated as an independently controlled variable so hardware and
numerical-format effects are never conflated. Every platform is benchmarked in
BF16, giving a like-for-like comparison that isolates raw hardware performance. The
three platforms with native FP8 tensor hardware—Gaudi2, H100, and GH200 (whose
Hopper GPU shares the H100's FP8 datapath)—are additionally benchmarked in FP8
(E4M3). The A100 (Ampere) lacks native FP8 tensor cores and, rather than using an
emulated non-comparable path, appears only in the BF16 comparison. Because Gaudi2
is reported in both BF16 and FP8, the throughput and energy benefit of quantization
is shown explicitly rather than mistaken for a hardware difference.

The two FP8 paths differ by backend. On Gaudi2, FP8 is produced through the Intel
Neural Compressor (INC): a one-time calibration pass records each linear layer's
maximum absolute activation per tensor and postprocesses these into per-tensor FP8
scaling factors, mapping each tensor independently to avoid overflow or precision
loss given FP8's narrow dynamic range. During inference, linear-layer weights and
activations are cast to FP8 (E4M3) and the matrix multiplications run in FP8 (W8A8)
on the Gaudi2 MME, with an FP8 KV cache; the LM head and non-linear operations
remain in BF16. On the NVIDIA Hopper platforms (H100 and GH200), vLLM applies FP8
online and dynamically from the BF16 checkpoint with an FP8 KV cache, requiring no
calibration step or quantized artifact. In every configuration the KV-cache
precision matches the compute precision—FP8 for FP8 runs, BF16 for BF16 runs—so
both comparisons remain like-for-like.

All experiments use a fixed workload of 512 input and 256 output tokens, with
request concurrency swept over 1, 8, 32, 64, and~128 to trace each platform from
the latency-bound regime to its throughput-saturation knee. An initial warmup pass
is discarded before measurement. On the NVIDIA (vLLM) platforms each configuration
is repeated three times and we report the median with run-to-run variability;
Optimum-Habana reports throughput aggregated over its own repeated iterations within
a single process. Token throughput and peak accelerator memory are recorded on every
platform, and TTFT, ITL/TPOT, and end-to-end latency on the NVIDIA platforms where
vLLM exposes the prefill/decode split. Board power is sampled live via
\texttt{nvidia-smi} on the GPUs and \texttt{hl-smi} on Gaudi2 and integrated into an
energy-per-token metric; each job sees only its own accelerator, so no co-located
workload contributes to the sampled power.
